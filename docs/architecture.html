Data Flow Sequences

How data moves through the system for key operations.

Flow Step-by-Step
Page Load (/contributors) 1. Server reads students_list from KV (or seeds from data/students.json)
2. Reads summary_cache:{period} from KV — if fresh, uses it
3. If no cache: calls getAllStudentSummaries() → reads each student's profile_cache:{username}
4. Cached students return immediately; uncached students get placeholder objects
5. Results sorted by scoreMergedPRs (merged PRs minus flagged ones)
6. SSR renders the full leaderboard grid
Refresh Button Click 1. Client calls POST /api/refresh?period=all
2. API checks if cache is fresh (< 5 min) — if so, returns early with "try again" message
3. If stale: calls getAllStudentSummaries() from profile caches (zero GitHub calls)
4. Writes result to summary_cache:all in KV
5. Calls revalidatePath('/contributors') to trigger Next.js ISR
6. Client calls router.refresh() to re-render
Individual Profile Refresh 1. Client calls POST /api/refresh?username=xyz
2. API fetches fresh data from GitHub: profile, PRs, issues
3. Writes to profile_cache:xyz in KV
4. Regenerates summary_cache for all/week/month periods
5. Revalidates the profile page and leaderboard
Daily Cron Job 1. Vercel Cron calls POST /api/refresh/incremental at 2:00 AM daily
2. updateStaleProfiles(5) finds the 5 oldest/uncached profiles
3. For each: fetches profile + PRs + issues from GitHub API
4. Writes to individual profile_cache entries
5. Regenerates summary caches for all/week/month
6. Revalidates Next.js paths
Flagging a PR 1. Admin clicks "Flag" on a PR in dashboard
2. Client calls POST /api/admin/flag
3. API adds entry to flagged_prs KV list
4. Calls invalidateSummaryCache() → sets all cache timestamps to epoch
5. Next page load rebuilds summaries with flagged PR excluded from scoring
Student Join Request 1. User fills form on /join page (GitHub username, year, campus)
2. Client calls POST /api/join-requests
3. API validates and adds to join_requests KV list as "pending"
4. Admin sees pending requests in dashboard
5. On approval: POST /api/admin/approve adds user to students_list
GitHub OAuth Login 1. User clicks login → redirected to /api/auth/github
2. API redirects to GitHub OAuth with client_id
3. GitHub redirects back to /api/auth/github/callback with code
4. Callback exchanges code for access token
5. Token stored in github_oauth_token cookie
6. Token used for GitHub API calls (higher rate limit)

Caching Architecture (3 Layers)

The system uses three caching layers to minimize GitHub API calls and ensure fast page loads.

Cache Layer KV Key Pattern TTL Purpose
1. Profile Cache profile_cache:{user} 30 days Stores a user's GitHub profile, all PRs, and all issues. One entry per student. Freshness check = 1 hour. This is the source of truth for the leaderboard — no GitHub API calls needed when reading from it.
2. Summary Cache summary_cache:{period} 24 hours Pre-computed leaderboard snapshots per time period (all, week, month, etc.). Built entirely from profile caches. Cooldown = 5 min between refreshes. This prevents re-processing 800+ students on every page load.
3. KV Store students_list, achievers_list, etc. No expiry Persistent domain data. Seeds from JSON files on first read. All admin CRUD operations update KV directly. In production uses Vercel KV (Redis), in dev uses local data/kv/*.json files.
